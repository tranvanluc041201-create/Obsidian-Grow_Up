# 1.9.1 宏定义优先级诅咒

## 🎯 题目 / 场景

宏定义计算结果与预期不符：

```c
#define DIV_MACRO(a, b)  a / b
#define ADD_MACRO(a, b)  a + b

// 调用
int result = DIV_MACRO(10 + 10, 2) * ADD_MACRO(2, 5);
```

---

## 💥 发生了什么

- **错误直觉**：认为结果是 `(10+10)/2 * (2+5) = 70`
- **实际结果**：`25`
- **文本展开**：`10 + 10 / 2 * 2 + 5` → `10 + (5 * 2) + 5` → `10 + 10 + 5 = 25`

---

## 🤔 我当时怎么想的

> "宏定义应该像函数一样，先计算参数，再执行宏体..."
> 
> **盲区**：以为宏定义有"参数传递"和"计算"的过程，实际上只是**纯文本替换**。

---

## 🔍 根因分析

### 表面原因
> 宏定义没有加括号，导致展开后运算优先级与预期不符

### 深层原理
> **宏定义的本质是文本替换**。预处理器不做任何计算，只是简单地把 `DIV_MACRO(10 + 10, 2)` 替换成 `10 + 10 / 2`。
> 
> 替换后的表达式 `10 + 10 / 2 * 2 + 5` 按照C语言运算符优先级：
> - `/` 和 `*` 优先级相同，从左到右：`(10 / 2) * 2 = 10`
> - `+` 优先级低：`10 + 10 + 5 = 25`

### 识别信号
> 凡是宏定义涉及运算的，**必须全程加括号**。
> - 参数加括号：防止参数是表达式
> - 整体加括号：防止宏作为子表达式时优先级错乱

---

## ✅ 正确规则与修正

```c
// ✅ 正确写法：参数和整体都加括号
#define DIV_MACRO(a, b)  ((a) / (b))
#define ADD_MACRO(a, b)  ((a) + (b))

// 展开结果：((10 + 10) / (2)) * ((2) + (5)) = 70 ✓
```

**规范参考**：《OpenHarmony 规范》规则 6.1：定义宏时，宏参数及整体都要使用完备的括号。

## 🔁 一题多变 / 横向关联（视情况补充）

### 同类变式
- **如果宏定义有多条语句？**
  ```c
  // ❌ 错误
  #define SWAP(a, b)  t = a; a = b; b = t
  
  // ✅ 正确：用 do { } while(0) 包裹
  #define SWAP(a, b)  do { t = a; a = b; b = t; } while(0)
  
  ```
### 🎯 问题场景：if-else 语句

```c
#define SWAP_BAD(a, b)  t = a; a = b; b = t;
#define SWAP_GOOD(a, b) do { t = a; a = b; b = t; } while(0)

// 使用场景
if (x > y)
    SWAP(a, b);  // ⚠️ 危险！
else
    do_something();
```

### ❌ 错误版本展开后

```c
if (x > y)
    t = a;   // ← if 只控制这一行
a = b;       // ← 这行永远执行！
b = t;       // ← 这行永远执行！
;            // ← 空语句，if 结束了
else         // ← 编译错误！else 找不到配对的 if
    do_something();
```

### ✅ do-while 版本展开后

```c
if (x > y)
    do { t = a; a = b; b = t; } while(0);  // ← 一个完整的语句
else
    do_something();
```

### 📊 对比表

|特性|`SWAP_BAD`|`SWAP_GOOD`|
|---|---|---|
|独立使用|✅ 正常|✅ 正常|
|`if(...) SWAP(x,y);`|⚠️ 危险（后半截逃逸）|✅ 安全|
|`if(...) SWAP(x,y); else ...`|❌ 编译错误|✅ 安全|
|需要分号|可选|**必须**（语法要求）|

### 🔥 额外好处

**1. 强制分号一致性**

```c
SWAP_GOOD(a, b)   // ✅ 正确，有分号
SWAP_GOOD(a, b);  // ✅ 也可以，while(0)吸收多余分号

// 实际上 do{}while(0); 是正常的C语句
```

### 📝 规则

```
多行宏必须用 do { } while(0) 包裹
    ↓
保证宏在控制流中表现像一个真正的函数调用
    ↓
避免分号问题和作用域问题
```

**现代C建议**：复杂逻辑直接用 `static inline` 函数，告别宏的坑。
- **如果宏定义有副作用？**
  ```c
  #define MAX(a, b)  ((a) > (b) ? (a) : (b))
  
  int x = 5, y = 3;
  int m = MAX(x++, y++);  // ❌ x++ 或 y++ 会被执行两次！
  ```

### 工程启示
> 在现代C开发中，能用 `inline` 函数就别用宏。宏的文本替换特性是历史遗留，现代编译器的优化能力已经让 `inline` 达到同样的性能，但类型安全、调试友好。
