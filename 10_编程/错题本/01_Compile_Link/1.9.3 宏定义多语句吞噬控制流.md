# 1.9.3 宏定义多语句吞噬控制流

## 🎯 题目 / 场景

多语句宏定义在if语句中导致控制流错乱：

```c
#define RESET_PERIPHERAL() \
    GPIO_A = 0; \
    GPIO_B = 0;

// 调用
if (hardware_error)
    RESET_PERIPHERAL();  // ❌ 只有第一条语句受if控制
```

**错误直觉**：只有当`hardware_error`为真时，才复位两个GPIO  
**实际结果**：`GPIO_A`随if执行，`GPIO_B`**永远**被置0

---

## 💥 发生了什么

**文本展开**：
```c
if (hardware_error)
    GPIO_A = 0;
GPIO_B = 0;  // ❌ 它是孤儿，不属于if！
```

**等价于**：
```c
if (hardware_error) {
    GPIO_A = 0;
}
GPIO_B = 0;  // 无条件执行
```

---

## 🤔 我当时怎么想的

> "宏定义就是简单的文本替换，多行也没问题..."
> "分号在宏定义里就是结束符..."
> **盲区**：不知道宏定义中的分号会成为展开后代码的一部分，破坏控制流结构。

---

## 🔍 根因分析

### 表面原因
> 多语句宏没有用`{}`或`do-while(0)`包裹，导致if只控制第一条语句

### 深层原理

**宏定义的语法陷阱**：
```c
// ❌ 错误：没有包裹
#define M() stmt1; stmt2;

if (cond)
    M();  // 展开：if (cond) stmt1; stmt2;

// ❌ 错误：用花括号
#define M() { stmt1; stmt2; }

if (cond)
    M();  // 展开：if (cond) { stmt1; stmt2; };  ← 多余的分号！
        
// ✅ 正确：do-while(0)
#define M() do { stmt1; stmt2; } while(0)

if (cond)
    M();  // 展开：if (cond) do { stmt1; stmt2; } while(0);
```

**为什么do-while(0)有效？**
- `do { ... } while(0)`是一个**完整语句**
- 末尾的分号成为语句结束符，语法正确
- `while(0)`确保只执行一次，不影响逻辑
- 即使没有分号调用（`M()`），也能编译通过

### 识别信号
> **多语句宏必须用do-while(0)包裹**，这是C语言的标准做法。

---

## ✅ 正确规则与修正

```c
// ✅ 标准写法：do-while(0)
#define RESET_PERIPHERAL() do { \
    GPIO_A = 0; \
    GPIO_B = 0; \
} while(0)

// 现在可以安全地用在if中
if (hardware_error)
    RESET_PERIPHERAL();  // 两条语句都受if控制

// 也可以用在else分支
if (cond)
    RESET_PERIPHERAL();
else
    do_something();  // ✅ 不会编译错误

// 甚至不用分号也能工作
if (cond)
    RESET_PERIPHERAL()  // ✅ 语法正确
```
