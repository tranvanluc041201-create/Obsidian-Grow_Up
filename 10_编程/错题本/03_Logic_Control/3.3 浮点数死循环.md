# 3.3 浮点数死循环

## 🎯 题目 / 场景

浮点数比较导致死循环：

```c
// ❌ 危险代码
for (float i = 0; i != 1.0; i += 0.1) {
    printf("%f\n", i);
}
```

**错误直觉**：循环10次，i从0到1.0  
**实际结果**：死循环（可能永远停不下来）

---

## 💥 发生了什么

**浮点数精度问题**：
- `0.1` 在二进制浮点中是**无限循环小数**（类似1/3在十进制中）
- 计算机存储的是**近似值**：`0.100000001490116119384765625`
- 每次累加误差累积，`i` 永远不等于精确的`1.0`

**可能的值**：
- 第10次累加后：`0.99999994` 或 `1.0000001`
- 永远不会等于 `1.0`

---

## 🤔 我当时怎么想的

> "0.1 + 0.1 + ... 十次应该等于1.0..."
> "浮点数计算应该足够精确..."
> **盲区**：不知道浮点数的二进制表示限制，很多十进制小数无法精确表示。

---

## 🔍 根因分析

### 表面原因
> 浮点数精度误差导致比较失败

### 深层原理

**浮点数表示**：
| 十进制 | 二进制浮点 | 是否精确 |
|--------|-----------|---------|
| 0.5 | 0.1 | ✅ 精确 |
| 0.25 | 0.01 | ✅ 精确 |
| 0.1 | 0.0001100110011... | ❌ 无限循环 |
| 0.2 | 0.001100110011... | ❌ 无限循环 |

**IEEE 754标准**：
- 单精度(float)：32位，约7位有效数字
- 双精度(double)：64位，约15位有效数字

**误差累积**：
```
0.1 (存储值) × 10 = 1.0000000149011612 ≠ 1.0
```

### 识别信号
> **浮点数比较黄金法则**：

> 1. 绝不使用`==`或`!=`比较浮点数
> 2. 使用误差范围：`fabs(a - b) < epsilon`
> 3. 循环计数用整数，最后转换为浮点数

---

## ✅ 正确规则与修正

```c
// ✅ 方法1：用整数计数（推荐）
for (int k = 0; k <= 10; k++) {
    float i = k / 10.0f;   // 关键：10.0f让除法变成浮点除法
    printf("i = %.7f\n", i);
}

// ✅ 方法2：使用误差范围比较
float i = 0.0f;
while (i < 1.0f - 1e-6) {  // 误差范围内
    printf("%f\n", i);
    i += 0.1f;
}

// ✅ 方法3：使用double（精度更高，但不解决根本问题）
for (double i = 0; i < 1.0; i += 0.1) {
    // 误差更小，但仍不精确
}

// ❌ 绝对禁止
for (float i = 0; i != 1.0; i += 0.1)  // 可能死循环
```

## 🔁 一题多变 / 横向关联（视情况补充）

### 同类变式
- **金钱计算**
  ```c
  float price = 0.1f;
  float total = price * 3;  // 0.30000001，不是0.3！
  // 金融计算应该用整数（分）或Decimal类型
  ```

- **三角函数**
  ```c
  float x = sin(M_PI);  // 理论上为0，实际是一个很小的数
  // x == 0 可能为假
  ```

### 横向关联
- **关联错题**：[[5.6] 浮点数比较精度问题]（同类问题）

### 工程启示
> **浮点数使用指南**：

> 1. 科学计算：可以接受小误差
> 2. 金融计算：**绝对不能用浮点数**，用整数（分）或专用Decimal库
> 3. 图形/游戏：误差可接受，但要注意比较方式
> 4. 嵌入式：可能根本不支持浮点硬件，尽量用定点数或整数
> **C语言建议**：
> - 默认用`double`（比float精度高）
> - 比较时用`<`、`>`，避免`==`、`!=`
> - 循环计数器用整数类型
