**ai**: 这篇 [[05-malloc-free]] 的核心是在“堆上的动态内存”里，练出一套可复制的安全习惯：**malloc 要判 NULL、大小要算对（含 `sizeof` 和字符串 `'\0'`）、用完要 free、free 后别再用**。并用一个“内存数据库（动态数组 + 记录含动态字符串）”把这些点串起来练到熟。[^1]

- 关键结论（新手必背）
- `malloc`：向堆申请连续内存，失败返回 `NULL`；每次分配都必须检查。[^2]
- 用完必须 `free`，否则会内存泄漏；`free` 后继续使用会变成 use-after-free。[^2]
- 最小正确模板（可直接抄）
- 正确申请：`malloc(n * sizeof(type))`，并在返回值为 `NULL` 时立刻处理/返回；用完 `free`，新手建议把指针置 `NULL` 防误用。[^3]

- 常见致命错误（最容易踩）
- `malloc(n)` 但按 `n` 个 `int` 用：忘了 `sizeof` 导致分配过小，进而越界写。[^4]
- 忘了判 `NULL`。[^4]
- 忘了 `free`（泄漏）。[^4]

- 实战训练（阶段 D）：把数据“存进内存结构”
- 目标：把读取到的记录存入动态数组 `Record[]`（或 `DB.items`），支持 list/find/del 等内存操作。[^1]
- 建议结构：`Record { char *name; char *number; }` + `DB { Record *items; size/cap }`，强调 `name/number` 都是动态分配出来的字符串。[^1]
- 两个“故意触发”的典型错误，用 Valgrind 识别：
- 字符串少分配 1 字节：`malloc(len)` 而不是 `malloc(len + 1)`，再 `strcpy` 或补 `'\0'`，会出现 `Invalid write`。[^1]
- 忘记释放 `record.name/record.number/db.items`，会出现 `definitely lost`。[^1]
- 修复标准：Valgrind 达到 `ERROR SUMMARY: 0 errors`，并在退出前释放所有分配的内存。[^1]

- 实战训练（阶段 E）：删除 + 扩容（realloc 正确姿势）
- 扩容必须用“临时指针”接 `realloc`：`tmp = realloc(...); if (tmp==NULL) 保留旧指针`，成功后再更新 `db->items/db->cap`。[^5]
- 必踩坑：直接 `db->items = realloc(...)`，一旦失败旧指针丢失，导致泄漏或状态损坏，甚至后续崩溃。[^5]
- 修复标准：realloc 失败时程序仍稳定、数据不丢、Valgrind 无泄漏。[^5]

- 也强调字符串边界要预留 `'\0'`（与动态字符串分配经常一起踩坑）。[^5]

### 3) [[03-pointers-null]]
- 聚焦 `NULL` 的语义与常见坑：
- `malloc` 失败会返回 `NULL`，必须检查，否则后续解引用可能崩。[^6]
- “未初始化指针”比 `NULL` 更阴险：`char *s; scanf("%s", s);` 会往垃圾地址写。[^7]
- 区分 `NULL` vs `'\0'`：`NULL` 是空指针给指针用；`'\0'` 是字符串结束符，存在字符数组里。[^8]

### 4) [[2.7 sizeof 指针的误解]]
- 专讲一个高频误区：函数形参里的数组会退化为指针，`sizeof(arr)` 变成 `sizeof(指针)`（4 或 8 字节），因此不能用 `sizeof(arr)/sizeof(int)` 算长度；应额外传 `length`。[^9]
- 虽然不是直接讲 `malloc`，但它解释了“为什么你以为算出来的长度对、实际却不对”的根因，间接影响分配大小与边界访问。

### 5) [[2.4 空指针解引用 (NULL Write)]]
- 用最小例子说明：`int *p = NULL; *p = 100;` 会因为写入地址 0（系统禁区）导致崩溃；修正是先让 `p` 指向合法内存（取地址或 `malloc`）。[^10]
- 和 “判 `NULL`” 强相关：不判就可能走到这种解引用路径。

## 汇总（把三类关键词串起来）
- **NULL**：`malloc` 可能失败返回 `NULL`，以及解引用之前要确保指针合法；未初始化指针也会导致类似灾难但更隐蔽（不是 `NULL` 是垃圾地址）。[^6][^7]
- **sizeof**：分配时用 `n * sizeof(type)`；别在函数里用 `sizeof(arr)` 误判数组长度（因为退化成指针）。[^2][^9]
- **use-after-free**：`free` 后继续使用属于 use-after-free，属于“更难查”的内存错误；实践里也推荐 `free` 后置 `NULL` 来降低误用风险。[^1][^2]
