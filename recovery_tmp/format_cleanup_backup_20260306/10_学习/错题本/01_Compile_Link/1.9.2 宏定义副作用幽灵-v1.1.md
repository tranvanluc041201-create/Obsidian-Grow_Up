## 🎯 Problem（问题）

宏定义中参数带副作用导致意外结果：

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int i = 5, j = 3;
int k = MAX(i++, j++);  // ❓ k=?, i=?, j=?
```

**错误直觉**：k=5, i=6, j=4  
**实际结果**：k=6, i=7, j=4

---

## 💥 发生了什么

**文本展开**：
```c
((i++) > (j++) ? (i++) : (j++))
```

**执行过程**：
1. 比较 `5 > 3`，条件为真，此时 `i=6, j=4`
2. 执行问号后的 `i++`，返回6，然后 `i` 再次自增变为7
3. `j++` 只执行了一次（在比较时）

**结果**：`k=6, i=7, j=4`

---

## 🤔 当时我是怎么想的

> "宏定义加了括号应该就安全了..."> "i++应该只执行一次..."> 
> **盲区**：不知道宏是**纯文本替换**，参数在宏体中出现几次就被替换几次，导致**多次求值**。

---

## 🔍 Analysis（分析）

### 表面原因
> 带副作用的表达式（i++）作为宏参数，导致多次自增

### 深层原理

**宏的本质是文本替换**：
```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))

MAX(i++, j++)
// 展开后：
((i++) > (j++) ? (i++) : (j++))
// i++ 出现了两次！
```

**即使加了括号也无济于事**：
- `((a) > (b) ? (a) : (b))` 中的括号解决了优先级问题
- 但解决不了**多次求值**问题

### 模式识别
> **函数式宏的危险**：
> - 参数在宏体中出现n次 → 会被求值n次
> - 如果参数有副作用（++/--/函数调用）→ 灾难> 
> **绝对禁止**：将带副作用的表达式传给函数式宏。

## ✅ Refix（修正）

```c
// ❌ 危险用法
int k = MAX(i++, j++);

// ✅ 安全用法1：先自增，再调用
i++; j++;
int k = MAX(i, j);

// ✅ 安全用法2：用内联函数代替宏（C99支持）
inline int max(int a, int b) {
    return (a > b) ? a : b;
}
int k = max(i++, j++);  // 现在i++只执行一次！

// ✅ 安全用法3：GCC扩展（语句表达式）
#define MAX(a, b) ({ \
    typeof(a) _a = (a); \
    typeof(b) _b = (b); \
    _a > _b ? _a : _b; \
})
// 先把参数存到临时变量，避免多次求值
```

## 💡 Insight（高阶认知）

### 同类变式
- **更隐蔽的例子**
  ```c
  #define SQUARE(x) ((x) * (x))
  int a = 5;
  int b = SQUARE(a++);  // ((a++) * (a++)) 未定义行为！
  ```

- **函数调用的副作用**
  ```c
  #define LOG(x) (printf("%s=%d\n", #x, x), x)
  int y = LOG(func());  // func()被调用两次！
  ```

### 工程启示
> **宏定义最佳实践**：> 1. **绝不**将带副作用的表达式传给宏> 2. 能用`inline`函数就别用宏（类型安全、无多次求值）> 3. 如果必须用宏，参数用大写提醒：>    ```c>    #define MAX(A, B) ...  // A、B提醒不要传复杂表达式>    ```> 
> **现代C/C++推荐**：
> - C++：用`template inline`函数
> - C99：用`inline`函数
> - 宏只用于：条件编译、简单的符号常量
