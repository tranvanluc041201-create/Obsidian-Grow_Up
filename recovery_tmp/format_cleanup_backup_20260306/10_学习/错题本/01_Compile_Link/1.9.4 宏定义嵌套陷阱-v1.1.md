## 🎯 Problem（问题）

宏定义嵌套调用时的优先级陷阱：

```c
#define F(a, b) a > b ? (a / b) : (a % b)
#define DF(k) 2 * k

int a = 14, b = 8;
int x = DF(F(b, a));  // ❌ 预期16，实际0
```

---

## 💥 发生了什么

**错误直觉**：先算内部`F(8,14)`得到8，再算`2*8=16`  
**实际结果**：`0`

**文本展开**：
```c
// DF(F(b,a)) 展开：
2 * b > a ? (b / a) : (b % a)

// 代入值：
2 * 8 > 14 ? (8 / 14) : (8 % 14)
// 16 > 14 为真，执行 8 / 14 = 0
```

---

## 🤔 当时我是怎么想的

> "宏应该从内向外展开，先算内层..."> "宏展开后应该保持运算优先级..."> 
> **盲区**：不知道宏是**纯文本替换**，展开后产生的新表达式优先级可能完全不同。

---

## 🔍 Analysis（分析）

### 表面原因
> 宏嵌套展开后，新表达式的运算优先级与预期不符

### 深层原理

**宏展开的本质**：
```c
#define F(a, b) a > b ? (a / b) : (a % b)
// 参数：a=b, b=a（注意参数名和值的混淆！）
// F(b, a) 展开为：b > a ? (b / a) : (b % a)
```

**DF(F(b,a)) 的展开过程**：
```c
// 第一步：展开F
DF(b > a ? (b / a) : (b % a))

// 第二步：展开DF（k = b > a ? ...）
2 * b > a ? (b / a) : (b % a)

// 问题：2 * b 先算，导致优先级错误！
// 实际：(2 * b) > a ? ...
```

### 模式识别
> **宏嵌套时，外层宏的参数如果包含运算符，会与外层表达式结合产生意外优先级**。

---

## ✅ Refix（修正）

```c
// ❌ 原始定义（危险）
#define F(a, b) a > b ? (a / b) : (a % b)
#define DF(k) 2 * k

// ✅ 修正：全程加括号
#define F(a, b) ((a) > (b) ? ((a) / (b)) : ((a) % (b)))
#define DF(k) (2 * (k))

// 现在展开：
// DF(F(b, a)) → (2 * ((b) > (a) ? ... : ...))
// 整个条件表达式被括号保护
```

## 💡 Insight（高阶认知）

### 同类变式
- **更复杂的嵌套**
  ```c
  #define ADD(a, b) a + b
  #define MUL(x) 2 * x
  
  int y = MUL(ADD(1, 2));  // 2 * 1 + 2 = 4，不是 6！
  // 展开：2 * 1 + 2
  ```

### 工程启示
> **宏定义铁律**：> 1. 宏参数**必须**加括号> 2. 宏整体**必须**加括号> 3. 复杂逻辑用 `static inline` 函数代替> 
> **嵌套宏的危险**：> 每层的括号都不能少，否则嵌套时优先级会层层叠加出错。
