## 🎯 Problem（问题）

函数内用`sizeof`计算数组长度失败：

```c
void print_array(int arr[]) {  // 或 int *arr
    int n = sizeof(arr) / sizeof(int);  // ❌ 永远得不到正确长度
    for (int i = 0; i < n; i++) {
        printf("%d ", arr[i]);
    }
}
```

---

## 💥 发生了什么

- **预期**：`n`等于数组元素个数
- **实际**：`n`等于1（64位系统可能是2）
- **原因**：数组传参退化为指针，`sizeof(arr)`得到的是指针大小（4或8字节）

---

## 🤔 当时我是怎么想的

> "sizeof(arr)/sizeof(int) 这个公式不是计算数组长度的标准做法吗..."
> "为什么在函数里就不对了..."
>
> **盲区**：不知道**数组传参退化为指针**，在函数内部`arr`已经是指针类型。

---

## 🔍 Analysis（分析）

### 表面原因
> 函数内使用`sizeof(arr)`得到的是指针大小，不是数组大小

### 深层原理

**数组退化（Array Decay）**：
```c
// 主函数中
int a[10];
sizeof(a);  // ✅ 40 (10个int)

// 函数参数中
void f(int arr[]) {  // 等价于 void f(int *arr)
    sizeof(arr);     // ❌ 4或8（指针大小）
}
```

**为什么C语言要这样设计？**
- 数组传递时**不拷贝整个数组**（效率考虑）
- 只传递首地址（指针）
- 因此丢失了长度信息

**解决方案对比**：
| 方法 | 优点 | 缺点 |
|------|------|------|
| 额外传长度参数 | 标准做法 | 调用者负担 |
| 用哨兵值（如-1） | 无需长度 | 数据不能含哨兵值 |
| 结构体封装 | 类型安全 | 稍复杂 |

### 模式识别
> **数组传参后，sizeof失效**。必须在传参前计算长度，或作为参数传入。

## ✅ Refix（修正）

```c
// ✅ 方法1：额外传长度参数（最常用）
void print_array(int arr[], int n) {
    for (int i = 0; i < n; i++) {
        printf("%d ", arr[i]);
    }
}

// 调用
int a[10] = {...};
print_array(a, 10);  // 必须传入长度

// ✅ 方法2：宏定义（只在定义数组的当前作用域有效）
#define ARRAY_SIZE(arr) (sizeof(arr) / sizeof((arr)[0]))

int main() {
    int a[10];
    int n = ARRAY_SIZE(a);  // ✅ 正确，得到10
    print_array(a, n);
}

// ✅ 方法3：结构体封装（类型安全）
typedef struct {
    int *data;
    int size;
} Array;

void print_array(Array arr) {
    for (int i = 0; i < arr.size; i++) {
        printf("%d ", arr.data[i]);
    }
}
```

## 💡 Insight（高阶认知）

### 同类变式
- **二维数组的sizeof**
  ```c
  void f(int arr[][10]) {
      sizeof(arr[0]);  // ✅ 40 (一行的大小)
      sizeof(arr);     // ❌ 仍然是指针大小
  }
  ```

> **C语言数组的本质弱点**：
> - 不携带长度信息
> - 容易越界
> - 现代语言（如C++的vector、Rust的slice）通过封装解决了这个问题> 
> **防御式编程**：
> - 总是同时传递数组和长度
> - 使用`const`修饰不修改的数组：`void f(const int arr[], int n)`
